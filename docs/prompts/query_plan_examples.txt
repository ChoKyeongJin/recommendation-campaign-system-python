# ─────────────────────────────────────────────────────────────
# 입력 패턴별 Query Plan few-shot 가이드
#
# 목적: "이런 구조의 입력이 들어오면 query_plan을 이렇게 채워라"를 LLM에 예시로 준다.
#
# 중요: 이 시스템에서 LLM은 SQL을 직접 만들지 않는다. LLM은 query_plan(JSON)만 만들고,
#       실제 SQL은 결정론 빌더가 생성한다. 따라서 아래 예시의 '기대 결과'도 SQL이 아니라
#       query_plan JSON이어야 한다. JSON 구조는 유저 프롬프트에 함께 실리는
#       [Fallback Rules Plan] 구조를 그대로 따른다(없는 필드/컬럼은 지어내지 말 것).
#
# 동작 규칙:
#   - '#'로 시작하는 줄은 이 파일 편집용 주석이며 LLM에 전달되지 않는다.
#   - '#'가 아닌 줄(아래 예시 본문)만 query_plan 시스템 프롬프트 뒤에 덧붙는다.
#   - 예시가 하나도 없으면(모두 주석이면) 프롬프트에 아무 것도 추가되지 않는다(기본 무동작).
#
# 편집 방법: 아래 '형식 견본'을 복사해 '#'를 떼고 실제 입력 패턴/기대 JSON으로 채운다.
#            여러 개를 이어서 추가할 수 있다.
#
# ── 형식 견본(주석 상태. 실제로 쓰려면 '#' 없이 다시 적으세요) ──
#
# [입력 패턴별 Query Plan 가이드]
# 아래는 특정 구조의 입력이 들어왔을 때 참고할 예시다. 입력이 아래 패턴과 유사하면
# 같은 방식으로 query_plan을 채워라. 값은 실제 질문에서 가져오고, 패턴에 없는 조건은
# 추가하지 마라.
#
# 예시 1)
# 입력 패턴: "<상품명> 구매한 고객 중 <등급> 등급"
# 기대 Query Plan(핵심 필드만, 구조는 Fallback Rules Plan을 따를 것):
# {
#   "intent": "find_user_segment",
#   "target_user": { "purchase_object": "<상품명>" }
# }
# ─────────────────────────────────────────────────────────────
#
# 주의(편집자용): 아래 '날짜 → 구매 날짜' 예시는 활성 상태다('#' 없는 줄).
#   - 이 규칙은 결정론(B)으로도 이미 강제된다:
#       · 상품명 추출(_sanitize_purchase_object)이 날짜 토큰을 상품 후보에서 제외한다.
#       · 절대 구매 날짜/기간은 _apply_purchase_date_filter → target_user.purchase_date 로 잡히고,
#         build_purchase_history_targets_sql_candidate 가 ORDER_DATE BETWEEN 으로 실제 SQL 에 반영한다.
#   - 따라서 이 few-shot 은 LLM 경로의 의도 설명 보조용이며, 최종 SQL 은 결정론 빌더가 보장한다.
#   - 연도 없는 'M월'(예: '3월')은 어느 해인지 모호해 구매 날짜로 잡지 않는다(오탐 방지). 연도를 포함하라.
#   - purchase_date 형식: {"from":"YYYYMMDD","to":"YYYYMMDD"} (예: 2024년 3월 → 20240301~20240331).

[입력 패턴별 Query Plan 가이드]
아래는 특정 구조의 입력이 들어왔을 때 참고할 예시다. 입력이 아래 패턴과 유사하면 같은 방식으로
query_plan을 채워라. 값은 실제 질문에서 가져오고, 패턴에 없는 조건은 추가하지 마라.

예시) 날짜/기간은 상품명이 아니라 '구매 날짜' 조건이다
입력 패턴: "<날짜/기간>에 <상품명> 구매/구입한 고객" (예: "2024년 3월에 기저귀 구매한 고객")
지침: 날짜/기간(연/월/일, 'N월', 'YYYY-MM' 등)은 purchase_object 에 넣지 마라.
      purchase_object 에는 상품/카테고리 이름만 넣고, 날짜/기간은 구매 날짜 조건(purchase_date)으로 분리하라.
      상품명 없이 날짜만 있으면 purchase_object 는 null 로 두고 날짜만 구매 날짜 조건으로 잡아라.
기대 Query Plan(핵심 필드만):
{
  "target_user": {
    "purchase_object": "기저귀",
    "purchase_date": { "from": "20240301", "to": "20240331" }
  }
}

# 주의(편집자용): 아래 '반기/분기 → 구매 날짜 창' 예시도 결정론(B)으로 이미 배선돼 있다
#   (_parse_half_or_quarter_period). 반기/분기는 반드시 연도 전(YYYY년) 뒤 창으로 뭉개지지 않게
#   '연 전체' 폴백보다 먼저 해석된다.

예시) 상반기/하반기·분기(반기/분기)는 구매 날짜 창이다 → 월 범위로 환산한다
입력 패턴: "YYYY년 상반기/하반기 … 구매/구입", "YYYY년 N분기(=N사분기) … 구매/구입"
지침: 반기/분기는 상품이 아니라 구매 날짜 조건(purchase_date)이다. 아래 월 범위로 환산해 담아라
      (모두 그 해 기준, 형식은 YYYYMMDD):
        · 상반기 = 1월1일~6월30일     (예: 2024년 상반기 → 20240101~20240630)
        · 하반기 = 7월1일~12월31일    (예: 2024년 하반기 → 20240701~20241231)
        · 1분기 = 1~3월, 2분기 = 4~6월, 3분기 = 7~9월, 4분기 = 10~12월
      연도가 없으면(그냥 '상반기'·'분기') 어느 해인지 모호하므로 잡지 마라(연도를 포함하라).
      그냥 '반기'(상/하 없이)나 숫자 없는 '분기'도 모호하므로 잡지 마라.
기대 Query Plan(핵심 필드만, 예: "2024년 하반기 구매 고객"):
{
  "target_user": {
    "purchase_date": { "from": "20240701", "to": "20241231" }
  }
}

# 주의(편집자용): 아래 '개수 제한 → TOP/LIMIT' 예시는 결정론(B)으로 이미 배선돼 있다.
#   - DBMS 매핑: CRMDW/CRMAN = MSSQL(T-SQL) → SELECT TOP N,  quadmax_sdz = MariaDB → LIMIT N.
#     실제 TOP/LIMIT 부착은 sql_guard(_apply_row_limit)가 대상 DBMS 방언에 맞춰 처리한다. LLM/파서는
#     개수만 plan.result_limit 에 담고, build_sql_template_response 가 그 값을 default_limit 로 넘긴다.
#   - 명시적 개수일 때만('N명만/상위 N명/최대 N명/N명으로 제한') 제한하고, 그 외 타겟 오디언스는
#     '전체가 나와야 한다'는 방침대로 무제한 유지한다(default_limit=None).
#   - 지표 랭킹(member_metric_ranking)·밀집지역(region_density)은 이미 SELECT TOP {top_n}을 쓰며,
#     sql_guard 가 기존 TOP 를 보존하므로 중복 TOP 가 붙지 않는다.
#   - 개수 제한은 조건이 아니므로 result_limit 는 target_user 가 아니라 plan 최상위에 둔다.

예시) "특정 개수만 뽑아라"는 개수 제한이다 → DBMS 에 맞는 TOP/LIMIT 로 반영한다
입력 패턴: "… 고객 <N>명만" / "상위 <N>명" / "… <N>건만 추출" (예: "구매 고객 100명만 뽑아줘")
지침: 요청된 개수는 타겟 조건(성별/등급 등)이 아니라 결과 행수 제한이다. 그 개수를 plan 에 담아라
      (랭킹이면 target_user.member_metric_ranking.top_n, 그 외 단순 개수 제한이면 result_limit).
      실제 SQL 은 대상 DBMS 에 맞춰 붙는다 — MSSQL(CRMDW/CRMAN)은 SELECT TOP N, MariaDB(quadmax_sdz)는
      LIMIT N. 개수를 상품명·수치 조건 등 다른 필드에 섞어 넣지 마라.
기대 Query Plan(핵심 필드만):
{
  "target_user": {
    "purchase_object": "기저귀"
  },
  "result_limit": 100
}
